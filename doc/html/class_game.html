Detailed Description

The Game class represents the whole game context and presents an API to others classes to interact with it.

The game can be in one of two modes: Database Generation mode and Play mode.

  • In Database Generation mode, the game will use a DBOutput instance to output a database of targeted actions taken. Play actions won't be recorded; for that you should use the Hearthstone Simulator Database Generator shipped with this engine.
  • In Play mode, the game is used to simulate Hearthstone, without any output.

The Game class should always be initialized using InitializeGlobals() at the beginning of any program.

After creating a Game object g, one should create a GameController object with g as constructor argument. One can also create a GameController with no arguments and use GameController::setGame() later with g as argument. See the sources of the GameThreadPool for an example of this last case.

Before entering the game loop, initPlayers() should be called with the thread in which the object will live once started. This way, players are living in the same thread than the game object.

Once everything is in place, emit a signal linked to the enterGameLoop() slot. This will start the game loop, i.e. the game will begin.

Warning
A Game object should always be encapsulated into a GameController that will take care of managing the thread in which the Game object and the players will live.
A Game object should always live in another thread than the main Qt event loop. This is managed automatically by the GameController it is linked to. The thread in which the Game object lives has to be a GameThread.
See also
GameController, GameThread, Hearthstone Simulator Database Generator

void Game::initPlayers ( )

Initialize the players of the game.

Move the players into the same thread than this Game object, then call their Player::initPlayer() methods.

Warning
This should always be called after the game has been moved to the thread in which it is going to be executed.
This should always be called before starting the game.
If this method is called from the moveTo thread, it has an undefined behavior.

void Game::nextTurn ( )

End the current player turn and begin the turn of the other player.

The sequence executed is the following:

  1. Unfreeze characters that lost their attack phase because of a Abilities::FROZEN condition
  2. Trigger a Event::AT_TURN_END with player set to the player whose turn is ended
  3. Switch current player and next player
  4. Call the Player::nextTurn() method of the new current player
Triggers:
Event::AT_TURN_END with
  • player : the player whose turn is ended
